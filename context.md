# Project Context

## Proven Patterns
- Для mobile-rebuild LORD в heavy-luxury стиле безопаснее держать весь редизайн только в `mobile/` и переписывать shared `assets/css/lord-rebuild-v1.css` + `assets/js/lord-rebuild-v1.js`, не трогая desktop; на главной карточки направлений должны иметь один однозначный CTA-переход, иначе пользователь воспринимает secondary action как поломку навигации.
- Если нужно вернуть desktop LORD как был и одновременно сохранить новый mobile rebuild, самый надёжный путь — держать mobile HTML в отдельной папке `mobile/`, а в `_worker.js` раздавать canonical routes по UA (`desktop -> публичные HTML`, `mobile -> /mobile/*` через внутренний fetch), скрывая прямой доступ к `/mobile/*` редиректом и добавляя `Vary: User-Agent, Sec-CH-UA-Mobile` + `Cache-Control: no-store` для HTML-вариантов.
- Если mobile-прод LORD продолжает ломаться из-за тяжёлого bootstrap/Tilda-слоя, самый надёжный выход — full-replace страниц на лёгкий static mobile-first каркас: локальные ассеты, один общий `assets/css/lord-rebuild-v1.css`, один общий `assets/js/lord-rebuild-v1.js`, без внешних шрифтов и без loader-цепочек; формы вести напрямую на `/api/lead`, а стабильность подтверждать WebKit-smoke локально и на `https://lord-event.ru`.
- Если после mobile-fix страница LORD на iPhone Safari остаётся «вечно загружающейся», нужно harden-ить сам `index.html`: ограничивать `t_onFuncLoad` по числу ретраев на mobile (`~40 попыток`, затем stub+continue), переводить не-критичные Tilda CSS (`animation/widgeticons/forms/slds/zero-gallery/katex`) в non-blocking `media="print" onload`, убирать внешние Google Fonts из head и отключать mobile-analytics (`Yandex`, `tilda-stat`) на главной.
- Если главная LORD на iPhone Safari всё ещё «бесконечно грузится» после mobile-hardening, первым делом проверять `assets/js/lord-main-v26.js`: main loader должен отключаться по `liteMode` (mobile/reduced-motion/lowCpu), а не только по `safeMode`; дополнительно нужно убирать `<link rel="preload" as="video" ... loader-main-0310.mp4>` из `index.html`, потому что даже отменённый mp4 preload даёт лишний хвост в mobile WebKit.
- Для video-loader на главной LORD нельзя закрывать экран по фиксированному `min/max` таймеру: надёжный контракт — ждать одновременно `video ended` и `window.load`, а прогресс-кольцо синхронизировать по `currentTime / duration` самого mp4; при этом в `index.html` обязательно поднимать `?v=` main CSS/JS.
- Для загрузчика главной LORD нельзя править только общий `site.js`: сама главная использует `assets/js/lord-main-v26.js` и `assets/css/lord-main-v26.css`, а чтобы loader-видео гарантированно стартовало, нужно положить mp4 в `assets/lord/`, добавить preload в `index.html` и поднять `?v=` у main CSS/JS.
- Для замены фона первого блока главной (`#rec1821410193`) нужно править `assets/css/lord-main-v26.css` (он реально подключён в `index.html`), а не `lord-overrides.css`; самый безопасный способ — append-only финальный override на `#rec1821410193 .t396__carrier` + обновление query-version CSS в `index.html`.
- Для блока Веры на Love стабильная схема: двухколоночный `vera-block` (`left=album`, `right=single profile card + short description`), при этом вторую карточку (`.vera-cred-card`) лучше скрывать в финальном tail-override, чтобы не появлялось пустое поле справа.
- Для компактных CTA-мессенджеров в hero на Love лучше использовать inline SVG-иконки в «облачных» мини-чипах (`::before/::after` как пузыри) вместо текстовых `TG/WA` — так кнопки читаются и на светлом фоне.
- Если нужно, чтобы первый блок Love начинался строго с самого верха экрана (под шапкой), лучший вариант: `position: fixed` для `body.page-love .site-header`, у `body.page-love > main` обнулить `margin/padding`, а hero-фон держать в `section-tight::before` с `top: 0`.
- Если на Love первый блок всё ещё стартует ниже шапки, нужен финальный tail-override на `body.page-love > main` с `padding-top: 0` и отрицательным `margin-top` от header-offset; фон hero при этом держать в `section-tight::before` с `top: -offset`.
- Когда на `/love` остаются «синие остатки» из старых каскадов, нужен единый финальный tail-блок в конце `styles.css` (hard-lock): обнулять `site-header` псевдослои, задавать светлый фон `body.page-love`, поднимать hero через `body.page-love > main` offset и ставить hero-image в `section-tight::before`; для блока Веры фиксировать двухколоночный layout (альбом слева, фото справа) в том же финальном блоке.
- Для финального фикса `/radost` при жалобах на «вставленные прямоугольники облаков» нужно проверять реальный alpha-канал ассета (`corners alpha`), подменять hero-cloud на contour-cut PNG и закреплять в HTML + tail-override (`object-fit: cover`, `overflow:hidden`, хаотичные `joy-media-mosaic-chaos`) в самом конце `assets/css/site.css`.
- Для hero в Tilda (`#rec1821410193`) градиентный текст `LORD` нужно рендерить через `-webkit-background-clip:text` + `-webkit-text-fill-color: transparent`, иначе на WebKit появляется «жёлтый прямоугольник» вместо букв.
- Самый безопасный способ менять визуал без поломки Tilda: добавлять финальные точечные правки в конец `assets/css/lord-overrides.css` отдельным hotfix-блоком с `body.lord-royal-v4 ...`.
- Пользовательские фоновые референсы лучше хранить локально в `assets/lord/` и подключать прямо в `#rec1821410193 .t396__artboard` как слой `url(...)` с отдельным затемняющим overlay для читаемости текста.
- Для запроса “сделай 8K” в hero удобнее перегенерировать те же имена ассетов (`hero-first-block-bg-8k.jpg/.webp`) из нового source-файла: CSS не меняется, фон обновляется сразу.
- Для тестовой быстрой замены без кэш-артефактов использовать новые суффиксы в именах (`hero-first-block-bg-<id>-8k.*`) и переключать `image-set(...)` в hotfix-блоке; для портретного source фиксировать фокус в `background-position` (например, `51% 46%`).
- Для глобального выравнивания «без швов» на Tilda-лендинге безопаснее добавлять финальный слой `LORD Final ...` в конец `assets/css/lord-overrides.css`: принудительно задавать единый `background-color` секциям, отключать `::before` bridge-градиенты и оставлять hero (`#rec1821410193`) исключением.
- Чтобы убрать «поочередную» догрузку при скролле, нужно перед скрытием loader форсировать lazy-фоны (`.t-bgimg[data-original]`, `[data-content-cover-bg]`) и расширять очередь preload background/image URL (через `waitForBackgroundImages` + `waitForCriticalImages`).
- Для Tilda-слайдера t728 расширение состава (новый участник) делается прямым добавлением `t-slds__item` + обновлением `aria-label "N из M"`; позицию соседнего заголовка лучше менять через JS-перестановку record-блоков (`insertBefore`) вместо хрупких отрицательных отступов.
- Для быстрой подмены hero на новый визуальный референс без риска кэша: создавать отдельные ассеты `hero-first-block-bg-<name>-8k.jpg/.webp` и переключать только `image-set(...)` в блоке `LORD Hotfix v5`.

## Proven Patterns (2026-03-11 release readiness check for split root)

## Proven Patterns (2026-03-11 full Pages release for split root)
- Для полного релиза split-root LORD через Cloudflare Pages надёжный порядок такой: (1) preflight на локали (`HTML_PARSE_OK` для root/mobile страниц и `node --check _worker.js`), (2) include-only staging bundle без мусора (`assets`, route-папки, `mobile`, `index.html`, `_worker.js`, `_headers`, `robots.txt`, `sitemap.xml`, `styles.css`, `script.js`), (3) deploy в `happiness-event` через `wrangler@4` под `node@22` с Account API Token из локального контекста, если обычный `wrangler login` уводит в чужой account, (4) post-deploy smoke на проде: desktop `/` должен содержать `lord-main-v26.css`, mobile `/` и `?view=mobile` — `lord-rebuild-v2.css` и `data-page="main-v2"`, `www` должен давать `301` на canonical host, прямой `/mobile/versions/v2/index.html` — `302` обратно на `/`, а `POST /api/lead` и `POST /api/rum` должны отвечать `{"ok":true}` и `204`.

## Proven Patterns (2026-03-11 rerun latest stable desktop release)
- Если пользователь просит «запустить последнюю стабильную desktop-версию», сначала нужно проверить, не работает ли она уже: desktop `/` должен содержать `lord-main-v26.css`, `index.html` должен совпадать с архивным stable snapshot, а desktop inner pages должны отвечать своими текущими stable title/CSS markers. После релиза нужно проверить, что desktop `/` остался на `lord-main-v26.css`, mobile `/` не съехал с `v2`, `www` всё ещё редиректит на canonical host, а `POST /api/lead` и `POST /api/rum` живы.

## Proven Patterns (2026-03-11 stable baseline fixation)
- Если текущий production state уже подтверждён как stable, этого мало держать только в памяти чата. Надёжнее сразу сохранять include-only snapshot во внешний архив (`.../_project-archive-YYYYMMDD/stable-release-YYYY-MM-DD-HHMM/site`), рядом класть manifest с deployment id, hash ключевых файлов и production markers, а в корне архива обновлять явный указатель вроде `LATEST_STABLE_RELEASE.md`.
